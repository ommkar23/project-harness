Review Spec

Sandboxed Coding Harness With Two Tools

This spec defines a coding harness that evaluates an agent inside a Vercel sandbox with a deliberately narrow interface. The agent has only two tools: text for user-facing messages and executePython for all work. Every file read, edit, search, install, test run, and repo operation happens through Python running in the sandbox.

Goals

Constrain the Agent

Make the execution surface easy to reason about by exposing only one working tool and one communication tool.

Run Safely

Execute generated code inside an isolated sandbox that contains the repo checkout and a prepared Python runtime.

Log Everything

Persist a complete append-only record of prompts, tool calls, tool results, and terminal outcomes for later inspection and scoring.

Architecture Review

What works

The diagram has the right control boundary: the harness is the orchestrator, the LLM only decides the next tool call, and the sandbox only executes generated code. That separation is clean and matches the two-tool model.

What to tighten

Context should not be treated as a generic shared blob. The harness should own the append order, prompt assembly, truncation, and summarization rules so the model never directly mutates history.

Recommended adjustment

Keep sandbox bootstrap and lifecycle outside the agent loop. The harness should prepare the repo and Python runtime first, then run iterative LLM and Python steps against that prepared environment.

Reference Architecture

System Prompt Head

Added once at context initialization. It describes the sandbox environment, tool contract, operating constraints, and safety rules.

Append-Only Context

The harness reads from this before each LLM call and appends new entries after each user message, assistant tool selection, and tool execution result.

Harness

The control plane. It receives user input from chat, assembles prompt context, sends context to the LLM, dispatches either text or executePython, captures outputs, and persists artifacts.

LLM

Receives prompt context from the harness and returns the next tool call: either user-facing text or Python code to execute.

Sandbox

Executes generated Python against the prepared repo and returns stdout, stderr, exit code, and other execution metadata to the harness.

Chat UI

Sends user requests into the harness and receives final or intermediary text responses.

Initialization

The harness creates context, prepends the system prompt head once, boots the sandbox, pulls the repo, prepares Python, and installs dependencies.

Loop

Read recent context, send it to the LLM, receive one tool call, execute it, then append the resulting record. The loop repeats until the harness emits a terminal user-facing response or hits a stop condition.

Append Order

Append user input, append assistant output or tool call, append tool result. This ordering from the diagram is correct and should stay deterministic.

Core Rules

Area Rule Notes
Tool surface The agent may call only text and executePython. No direct shell tool, grep tool, edit tool, or file search tool exists.
User communication text is the only mechanism for user-facing output. Intermediary updates and final answers both flow through this tool contract.
Work execution executePython runs inline Python in the sandbox repo root. Python may use stdlib, installed packages, and subprocess.
Operational preference Prefer native Python APIs over shell commands wherever practical. Use pathlib, json, re, urllib, and related stdlib modules first.
Sandbox setup The host creates the sandbox, pulls the repo, prepares Python, and installs dependencies before the loop starts. The model should not be responsible for environment bootstrap.
Context ownership The harness owns prompt assembly and append order. The model can observe context through prompts, but cannot rewrite prior entries.
Context The harness stores an append-only context log for the entire run. Older entries can be summarized for prompts, but raw history remains preserved on disk.

Execution Flow

1

Bootstrap the sandbox

Create a fresh Vercel sandbox, clone or pull the target repository, prepare Python, install repo dependencies, and record the environment summary that will be exposed to the agent.

2

Initialize context head

Create the append-only context log and prepend the one-time system prompt head, including the repo AGENTS.md, the sandbox environment contract, and the explicit two-tool constraint.

3

Read context and generate the next action

The harness reads the current context window before each LLM call. The model then responds with either a text call or an executePython call. If it needs to inspect files, search code, run tests, edit content, or invoke git, it must do that through Python.

4

Execute inline Python

The harness runs the Python snippet inside the sandbox using the prepared Python runtime and working directory. The snippet can read and write files directly and can use subprocess when native Python is not the right interface.

5

Append results to context

Every user message, model output, tool call, tool result, and stop condition is appended to the run log in deterministic order: user input, assistant action, tool result. The next prompt sees a recent working window plus any generated summary of older entries.

6

Stop and persist artifacts

When the task is complete, the harness stores the final transcript, tool execution metadata, summaries, and stop reason. The sandbox is then destroyed.

Tool Contract

text(message)

  • Used only to convey a message to the user.
  • Must not be used to simulate file edits or fake tool output.
  • Can be used for progress updates and final answers.

executePython(code)

  • Executes inline Python in the sandbox repository root.
  • May use stdlib, installed project dependencies, and subprocess.
  • Handles file reads, file writes, searches, installs, tests, and repo operations.

Environment Contract

What the agent should know

  • Absolute repo path inside the sandbox.
  • Python runtime details when relevant.
  • Default working directory.
  • Available network policy and install policy.
  • Timeout, stdout, and stderr truncation limits.

What the host should handle

  • Sandbox creation and teardown.
  • Repo checkout or clone.
  • Python setup and baseline dependency installation.
  • Injecting the environment summary into the prompt.
  • Capturing execution metadata for scoring and auditability.

Append-Only Context Model

Entry Schema

  • id
  • timestamp
  • type: system, user, assistant, tool_call, tool_result
  • content
  • metadata: duration, exit_code, truncation, files touched

Prompting Strategy

  • Preserve the raw log as append-only history on disk.
  • Use a recent context window for active reasoning.
  • Summarize older entries when token pressure increases.
  • Never mutate or delete historical entries from the run artifact.

Run Artifact Requirements

Artifact Purpose Minimum contents
Run summary High-level scan of the session Run id, timestamps, stop reason, completed steps, overall outcome
Context log Full audit trail All append-only entries in order
Tool execution records Debugging and scoring Python source, stdout, stderr, exit code, duration, truncation flags
Environment snapshot Reproducibility Repo ref, sandbox config, Python version, and install inputs

AGENTS.md Requirements

The repo-level AGENTS.md should explicitly describe the two-tool model, the sandbox environment, the Python-first operating preference, subprocess availability, safety rules for destructive commands, and the expectation that all repo interaction occurs through executePython.

Open Decisions

Likely decisions to finalize

  • Whether the sandbox is per run, per task, or reused across a bounded session.
  • Whether extra package installs are unrestricted, blocked, or allowlisted.
  • Whether network access is open, restricted, or disabled after bootstrap.
  • Whether file-change diffs are stored after each executePython call.

Recommended defaults

  • Fresh sandbox per run.
  • Host installs baseline repo dependencies before the loop starts.
  • Per-call timeout and output truncation.
  • Store raw transcript plus a compact summary.

This is a product and runtime spec for review, not an implementation log. It is meant to make the proposed harness behavior concrete before code changes begin.
