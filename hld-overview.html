High-Level Design

Project Harness: Chat-Backed Sandbox Coding System

This project is a lightweight chat harness for sandboxed repository work. Its job is to let the model inspect and change a target Git repository through inline Python executed inside Vercel Sandbox, while keeping the interaction stream visible in the Next.js UI.

System Intent

Primary Goal

Provide a controlled environment for repository-aware coding tasks without giving the model unrestricted host access.

Current Use Case

Answer user requests by running Python against a cloned repository, then stream the result back through the chat UI.

Operating Model

Accept a chat message, convert it into model context, execute a narrow server-side tool when needed, and reuse the sandbox for the life of the chat session.

Architecture

Execution Flow

1

User Request

The user describes a repository task in the chat UI and the browser posts the message to the chat route.

2

Model Context Build

The route strips completed tool parts, converts the conversation to model messages, and prepends the sandbox summary.

3

Tool Execution

When the model needs repo access, it calls `executePython`, which runs a temp Python file inside the sandbox and returns stdout, stderr, and exit status.

4

Sandbox Reuse

The session store keeps the sandbox id in memory so later messages in the same chat can continue from the same repository state while the server stays up.

5

UI Streaming

The response is streamed back as AI SDK UI messages, so text answers and tool output stay in the same conversation.

Current Boundaries

What It Covers Today

  • Sandboxed repository access through inline Python
  • Single-tool chat orchestration
  • Gemini-backed response generation
  • Per-chat sandbox reuse while the process lives
  • Braintrust telemetry for chat and execution traces

What It Does Not Cover Yet

  • Persistent sandbox sessions across server restarts
  • Multi-tool or multi-agent workflows
  • Persistent chat memory outside the current session
  • Production orchestration or scheduling
  • General-purpose model routing or provider comparison

Design Summary

In HLD terms, this is a narrow chat-and-sandbox system. The core value is not raw model access, but the ability to safely execute repository work through a controlled tool surface and keep the interaction visible in the conversation.

This page intentionally avoids implementation details and code structure. It is meant to explain the system as a product and architecture concept.
